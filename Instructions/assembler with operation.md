
 Assembler က assembly ကို hex တွေအဖြစ် ပြောင်းလိုက်ရင် အချို့ instruction တွေကို  ကြိုတင်တွက်ချက်မှု (Evaluation) တွေကို လုပ်ပေးတယ်
ဒီအခြေအနေကို ကွန်ပြူတာသိပ္ပံမှာ **Constant Folding** သို့မဟုတ် **Compile-time Evaluation** လို့ ခေါ်ပါတယ်

---

**Constant Folding** သို့မဟုတ် **Compile-time Evaluation**  ဘယ်လိုဖြစ်မလဲဆို



 1. တန်ဖိုးအားလုံးက ကိန်းသေ (Constants) ဖြစ်ရတယ်

Assembler က code ကို compile လုပ်နေတဲ့ အချိန်မှာတင် အဖြေကို ရှာလို့ရမယ့် တန်ဖိုးမျိုး ဖြစ်ရပါမယ်။

- **အလုပ်လုပ်မည့် အခြေအနေ:** `0x2a2a2a2a2a2a2a2a ^ 0x68732f6e69622f`
    _(ဒီနေရာမှာ ကိန်းဂဏန်းနှစ်ခုလုံးက အသေ သတ်မှတ်ထားတာ ဖြစ်လို့ Assembler က compile လုပ်ရင်းနဲ့ နောက်ကွယ်မှာ `0x42590544434805` ဆိုပြီး ချက်ချင်း တွက်ထုတ်ပေးလိုက်ပါတယ်။)_
    
- **အလုပ်မလုပ်မည့် အခြေအနေ:** `mov rax, rbx ^ 0x68732f6e69622f`
    _(ဒီလိုမျိုး Register တစ်ခုနဲ့ သွားတွက်ခိုင်းရင်တော့ Assembler က ကြိုမတွက်ပေးနိုင်ပါဘူး။ ဘာလို့လဲဆိုတော့ `rbx` ထဲမှာ ဘာတန်ဖိုး ရှိနေမလဲဆိုတာ ပရိုဂရမ်တကယ် run တဲ့အချိန် (Runtime) ကျမှပဲ သိနိုင်လို့ ဖြစ်ပါတယ်။)_
    

---


 2. Assembler က နားလည်တဲ့ Operator တွေ ဖြစ်ရတယ်

ကျွန်တော်တို့ သုံးနေတဲ့ GNU Assembler (`as`) သို့မဟုတ် `nasm` တွေက သင်္ချာနဲ့ Logic ပိုင်းဆိုင်ရာ Operator တော်တော်များများကို ကြိုတင်တွက်ချက်ပေးနိုင်စွမ်း ရှိပါတယ်။

 `mov rax, [Expression]` နေရာမှာ အခုလို အခြေအနေမျိုးတွေ ရေးရင်လည်း Assembler က ကြိုတွက်ပေးမှာ ဖြစ်ပါတယ်

|**Operator အမျိုးအစား**|**ဥပမာ ရေးထုံး**|**Assembler ကြိုတွက်ပေးမည့် ပုံစံ**|
|---|---|---|
|**Bitwise XOR (`^`)**|`0x10 ^ 0x20`|`0x30`|
|**Bitwise AND (`&`)**|`0xFF & 0x0F`|`0x0F`|
|**Bitwise OR (`\|`)**|`0x10 \| 0x01`|`0x11`|
|**အပေါင်း / အနှုတ် (`+`, `-`)**|`100 + 50 - 20`|`130`|
|**အမြှောက် / အစား (`*`, `/`)**|`10 * 8`|`80`|
|**Bit Shifting (`<<`, `>>`)**|`1 << 4`|`16` (`0x10`)|
