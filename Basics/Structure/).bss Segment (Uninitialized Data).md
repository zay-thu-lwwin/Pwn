BSS က Block Started by Symbol (သို့) Better Save Space လို့ အဓိပ္ပာယ်ရပါတယ်
 `.data` လိုပဲ data segment ထဲမှာရှိတယ်
Uninitialized သို့မဟုတ် zero-initialize global/static variables တွေအတွက် နေရာ
Program စတင်တဲ့အခါ kernel က ဒီနေရာကို zero တွေနဲ့ ဖြည့်ပေးတယ် (အလိုအလျောက်)
Binary file (disk) ထဲမှာ နေရာမယူဘူး → memory မှာမှ runtime မှာ allocate လုပ်တယ်
`bss` က `.data` ရဲ့အပေါ်မှာ (သို့) ကပ်လျက်မှာရှိတယ် Permission က read-write (same as .data)


BSS ဆိုတာ uninitialized static variables တွေအတွက်၊ program start မှာ zeros ဖြည့်ပေးတယ်၊ read/write လုပ်လို့ရတယ်၊ binary file ထဲမှာ space မယူဘူး

```c
// ALL DATA STORED IN BSS SEGMENT
// ============================================

// 1. UNINITIALIZED GLOBAL VARIABLES
int global_int;                    // BSS → value: 0
float global_float;                // BSS → value: 0.0
char global_char;                  // BSS → value: '\0'
double global_double;              // BSS → value: 0.0
void *global_ptr;                  // BSS → value: NULL

// 2. UNINITIALIZED GLOBAL ARRAYS
int global_arr[100];               // BSS → all zeros
char global_str[50];               // BSS → all '\0'
float global_float_arr[20];        // BSS → all 0.0

// 3. UNINITIALIZED STATIC VARIABLES (global scope)
static int static_global_int;      // BSS (file scope)
static char static_global_char;    // BSS (file scope)

// 4. UNINITIALIZED STRUCTURES (global)
struct Point {
    int x;
    int y;
    char name[10];
};
struct Point global_point;         // BSS → x=0, y=0, name all '\0'

// 5. UNINITIALIZED GLOBAL POINTERS
int *global_ptr_ptr;               // BSS → NULL
char **global_ptr_ptr_ptr;         // BSS → NULL

// 6. UNINITIALIZED FUNCTION POINTERS
void (*func_ptr)(void);            // BSS → NULL

void function() {
    // 7. STATIC LOCAL VARIABLES (uninitialized)
    static int local_static_int;   // BSS (not stack!)
    static char local_static_arr[30]; // BSS
    static float local_static_float;   // BSS
    
    // 8. STATIC LOCAL POINTERS
    static int *local_static_ptr;  // BSS → NULL
    
    // ⚠️ NON-static local variables go to STACK, not BSS!
    int local_int;                 // STACK (garbage value)
}

// 9. GLOBAL ARRAY with partial initialization (rest goes to BSS)
int partial_arr[5] = {1, 2};       // First 2 in DATA, remaining 3 in BSS
// Actually: This entire array goes to DATA, but conceptually 
// uninitialized elements are zero. Modern compilers put entire array in DATA.

// 10. ZERO-INITIALIZED GLOBAL (compiler might put in BSS)
int zero_init = 0;                 // BSS (optimization, usually BSS not DATA)
static int static_zero = 0;        // BSS
```

##### not in bss

| Data Type                       | Storage            | Why not BSS?                 |
| ------------------------------- | ------------------ | ---------------------------- |
| `int x = 5;` (global)           | **DATA**           | Has initial value (non-zero) |
| `int x;` (local)                | **STACK**          | Local, not static            |
| `static int x = 5;`             | **DATA**           | Initialized static           |
| `const int x = 5;`              | **TEXT/rodata**    | Read-only                    |
| `char *p = "hello";`            | **DATA** (pointer) | Initialized pointer          |
| `malloc()` memory               | **HEAP**           | Dynamic allocation           |
| `char arr[] = "hello";` (local) | **STACK**          | Local array                  |

##### BSS vs DATA Comparison

|Aspect|BSS|DATA|
|---|---|---|
|**Initial value**|0 (implicit)|Explicit (non-zero)|
|**Binary file size**|No space (only size)|Takes space|
|**Example**|`int g;`|`int g = 5;`|
|**Read/Write**|✅ Yes|✅ Yes|
|**Static locals**|`static int x;`|`static int x=5;`|
|**Zero-init optimization**|`int x = 0;` → BSS|`int x = 1;` → DATA|

**Important**: `int x = 0;` (global) usually goes to **BSS**, not DATA! (Compiler optimization)

### Exploitation

| Feature            | BSS                                               |
| ------------------ | ------------------------------------------------- |
| **Can overwrite?** | ✅ Yes (read-write)                                |
| **ASLR**           | Usually no ASLR (fixed address in many binaries)  |
| **Useful for**     | Storing shellcode, fake vtable, function pointers |
| **Initial value**  | All zeros (good for NULL-terminated strings)      |
| **Size limit**     | Can be large (but determined at compile time)     |

#### Large buffer remains without increasing the binary size

ဆိုလိုတာက programmer တွေက ကြီးမားတဲ့ buffer တွေ (eg. 10KB, 1MB) ကို `.bss` ထဲ သတ်မှတ်လေ့ရှိတယ် - binary size မပူရလို့
```c
char input_buffer[4096];  // .bss (4KB)
char heap_buffer[4096];   // heap (4KB)
```
Exploitation အတွက် - ဒီ large `.bss` buffer ကို shellcode (သို့) ROP chain သိမ်းဖို့ သုံးလို့ရတယ်
```c
// Vulnerability: stack buffer overflow
char stack_buf[64];
gets(stack_buf);  // overflow!

// Attacker က stack_buf ကနေ .bss ထဲက input_buffer ဆီ return address ကို ပြောင်းလိုက်မယ်
// input_buffer ထဲမှာ shellcode ထည့်ထားပြီးသား
```

####  Use-After-Free

Uninitialized ဖြစ်လို့ Use-After-Free လိုမျိုး bug တွေပေါ်လွယ်တယ်

```c
static char *ptr;  // .bss → zero-initialized (NULL)

void init() {
    if (!ptr) {
        ptr = malloc(100);
    }
}

void use() {
    strcpy(ptr, "data");  // ptr NULL ဖြစ်နေရင် crash
}
```

Exploitation - `.bss` က zero ဖြစ်တဲ့အတွက် programmer တွေ initialization မေ့တတ်တယ် → NULL pointer dereference သို့မဟုတ် uninitialized use ဖြစ်တယ်
တခါတလေ `.bss` ထဲက variable ကို read before write လုပ်မိရင် ရှိပြီးသား တန်ဖိုး (old data from previous function call) ကို ပြန်သုံးမိတယ် → information leak ဖြစ်နိုင်တယ်

#### Binary size

Binary size သေးတဲ့အတွက် သိပ်မသိသာဘူး၊ ဒါပေမဲ့ runtime မှာ ရုတ်တရက်ကြီးလာနိုင်တယ်
`.bss` က disk မှာ သေးပေမယ့် run လိုက်တဲ့အခါ memory ထဲမှာ အပြည့်နေရာယူတယ်
ဒါက exploitation အတွက် ဘာလို့ကောင်းတာလဲ
-  binary ကို download လုပ်ကြည့်လိုက်တယ် → size သေးတယ် → ဂရုမစိုက်မိဘူး
- ဒါပေမဲ့ run လိုက်ရင် memory ထဲမှာ 100MB buffer ရှိနေတယ် (`.bss` ထဲ)
- ဒီ buffer ကိုသုံးပြီး large ROP chain (သို့) stage 2 shellcode ထည့်လို့ရတယ်
